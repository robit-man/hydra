## Implementation Checklist
1. Define shared visual tokens (typography, accent, status colors, badge style) and apply to both apps.
2. Align peer card fields and terminology (`address`, `network`, `transport`, `last seen`, `status`).
3. Add equivalent manual add + QR scan/import actions on NoClip to match Hydra capabilities.
4. Surface transport path (`NKN`, `cloudflare`, `relay/local`) in both peer/chat interfaces.
5. Ensure parity on mobile and desktop layouts for peer list + chat + action buttons.

## Success Metrics
- Hydra and NoClip show consistent peer identity and transport status terminology.
- Users can perform the same connect flow (manual add or QR) from either frontend.
- Peer and chat affordances map one-to-one (`Chat`, `Call`, `Video`, `Bridge/Sync` where applicable).
- Responsive behavior remains functional on narrow/mobile viewports in both apps.

## Potential Failure Modes And Preventions
- Failure mode: style divergence reappears due duplicated inline styling.
  Prevention: centralize tokens and avoid component-local hardcoded colors/fonts.
- Failure mode: one app allows actions hidden/unsupported in the other.
  Prevention: publish a shared action matrix and gate UI buttons by feature availability.
- Failure mode: QR payload variants parse differently across apps.
  Prevention: one shared parser contract for `?peer=`, `?hydra=`, `?noclip=` with strict validation.
- Failure mode: mobile layout regressions hide critical controls.
  Prevention: add viewport regression checks for peer modal and chat/friends panels.
