## Implementation Checklist
1. Add manual peer entry fields on NoClip for `hydra.<hex64>` and `noclip.<hex64>` inputs.
2. Add QR encode/decode for peer invites on NoClip (matching Hydra scanner behavior).
3. Support import link forms: `?peer=`, `?hydra=`, `?noclip=` with canonicalization.
4. Add explicit user feedback states: `invalid`, `pending`, `connected`, `duplicate`.
5. Route accepted peer entries into discovery/peer stores without requiring friend request APIs.

## Success Metrics
- A user can connect Hydra↔NoClip by scanning a QR code or pasting an address manually.
- Invalid addresses are rejected with actionable validation errors.
- Manual peer entries survive refresh and are visible in peer/friend chat lists.

## Potential Failure Modes And Preventions
- Failure mode: malformed QR payloads poison local peer state.
  Prevention: strict parse/validate and ignore unknown payload schema versions.
- Failure mode: manual peer list diverges from discovery peers.
  Prevention: merge by canonical key with source metadata (`manual`, `discovery`, `friend`).
- Failure mode: untrusted links auto-connect silently.
  Prevention: require explicit user confirmation before persistent addition.
- Failure mode: duplicate UX controls across panels cause conflicting state.
  Prevention: centralize peer-add actions behind one service API in each frontend.
