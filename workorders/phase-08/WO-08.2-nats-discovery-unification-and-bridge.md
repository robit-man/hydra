## Implementation Checklist
1. Adopt one shared subject shape:
- Presence: `discovery.<room>.presence`
- Direct discovery DM: `discovery.<room>.dm.<pub>`
2. Add a compatibility listener for legacy subject forms during migration window.
3. Make discovery room explicit and persisted on both apps (`nexus` default).
4. Ensure discovery payloads carry `meta.network` with strict values `hydra` or `noclip`.
5. Prevent duplicate discovery stacks (Hydra has `peerDiscovery.js` and `nats.js` overlap today).

## Success Metrics
- Hydra peers and NoClip peers discover each other in the same room without custom per-app patches.
- No duplicate peer events from parallel discovery clients.
- Discovery reconnects cleanly and does not block NKN direct messaging.

## Potential Failure Modes And Preventions
- Failure mode: room mismatch isolates peer sets.
  Prevention: one default room constant + visible UI control + persisted state.
- Failure mode: event storms from dual discovery clients.
  Prevention: single active discovery client with explicit teardown/ownership.
- Failure mode: wrong subject convention causes one-way discovery.
  Prevention: temporary dual-subscribe compatibility layer with telemetry warnings.
- Failure mode: NATS outage breaks peer UX.
  Prevention: degrade gracefully to manual/NKN paths and mark discovery status as degraded.
