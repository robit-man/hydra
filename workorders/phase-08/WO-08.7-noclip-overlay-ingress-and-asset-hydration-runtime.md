## Implementation Checklist
1. Add `HydraOverlayIngress` service to translate Hydra envelopes into OverlaySubsystem operations.
2. Resolve target mapping (`objectId`/`sessionId`/`overlayId`/`itemId`) before mutating scene state.
3. Support hydration of:
- theater stream updates (audio/video),
- model URL or GLB asset replacement,
- pointcloud geometry updates,
- live position/move updates.
4. Emit ingress acknowledgments and error reasons back to Hydra bridge sessions.
5. Apply permission/gating checks before loading or mutating protected overlay assets.

## Success Metrics
- A Hydra command/resource packet can create or update a specific overlay item in NoClip.
- Theater overlays render remote media and attach correct spatial audio streams.
- GLB/pointcloud updates can be applied without reloading the full scene.
- Rejected operations provide explicit error events (no silent no-op behavior).

## Potential Failure Modes And Preventions
- Failure mode: missing target mapping applies updates to wrong overlay item.
  Prevention: require deterministic resolver with strict ID matching and reject ambiguous targets.
- Failure mode: asset hydration races leave partially updated objects.
  Prevention: apply staged update pipeline (validate -> load -> swap -> commit).
- Failure mode: unauthorized overlays are mutated through realtime channels.
  Prevention: enforce access checks in ingress runtime and backend overlay subscribe path.
- Failure mode: repeated realtime updates saturate render loop.
  Prevention: throttle/coalesce high-frequency position updates and cap queue depth.
