# WO-11.6 URL Invite Bridge Auto-Target And Context Hydration

## Objective
Convert NoClip URL invite ingress from passive alert-only behavior into executable bridge targeting so invite links can automatically configure a `NoClipBridge` peer target and optional object/session context in Hydra without manual node edits.

## Files To Touch (Hydra)
- `docs/js/main.js`:
  - extend bridge-target orchestration to accept object/session context and optional preferred bridge node around lines 876-968.
  - replace invite URL handling logic with event-driven bridge-target application around lines 1553-1630.
- `workorders/README.md`:
  - add phase-11 index entry for `WO-11.6`.
- `docs/HYDRA_NOCLIP_MARKETPLACE_RUNBOOK.md`:
  - add operator validation for URL invite auto-target and context hydration.
- `docs/HYDRA_NOCLIP_INTEROP_ROLLOUT.md`:
  - extend frontend sanity checks with URL invite validation.

## Implementation Checklist
1. Add bridge-target context normalization (`sessionId`, `objectUuid`, `overlayId`, `itemId`, `layerId`) in main orchestration path.
2. Allow bridge-target requests to include optional preferred bridge node id.
3. Persist context fields to `NoClipBridge` config (`interopTarget` + direct fields) when provided.
4. Replace invite URL `alert` flow with dispatch to `hydra-noclip-bridge-target`.
5. Expand invite parser to accept alias params (`peer`, `target`, object/session/overlay/item/layer keys).
6. Preserve sanitized URL cleanup after invite ingestion.
7. Update runbook + rollout checks for invite auto-apply behavior.

## Success Metrics
- Opening Hydra with `?noclip=<pub>` auto-configures an existing or newly created `NoClipBridge` node target.
- Invite URLs containing object/session keys hydrate bridge config target context without opening settings.
- Invite processing no longer depends on a blocking browser alert.
- URL query params used for invite ingress are removed after processing.

## Potential Failure Modes And Preventions
- Failure mode: malformed invite pubkey mutates node state.
  Prevention: strict pubkey parser and `invalid_noclip_target_pub` fast-fail.
- Failure mode: invite context lost after target assignment.
  Prevention: write both direct config fields and `interopTarget` object in one patch.
- Failure mode: invite keeps re-triggering on reload due to stale query params.
  Prevention: remove all known invite params via `history.replaceState`.
- Failure mode: wrong bridge node gets updated in multi-bridge graphs.
  Prevention: respect explicit invite `bridgeNodeId/nodeId` before fallback node selection.
