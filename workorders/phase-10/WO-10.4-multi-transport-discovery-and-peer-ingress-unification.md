## Implementation Checklist
1. Canonicalize address prefixes and transport candidate shapes in both frontends.
2. Support symmetric manual add and QR ingest for `hydra.*` and `noclip.*` identities.
3. Keep NKN as primary DM path with NATS fallback and explicit source tagging.
4. Propagate selected transport and candidate freshness through presence updates.
5. Add deterministic conflict resolution when multiple candidates are present.

## Success Metrics
- Hydra and NoClip peers can connect via manual paste or QR in both directions.
- Discovery state shows the same selected transport on both frontends.
- Failed NKN DM attempts fall back to NATS without message loss.
- Peer cards display network + transport + freshness consistently.

## Potential Failure Modes And Preventions
- Failure mode: duplicate peer identities from prefix/format mismatch.
  Prevention: one canonical address normalizer with strict lowercasing and prefix rules.
- Failure mode: stale candidate chosen despite fresh alternative.
  Prevention: candidate scoring with stale penalty and last-verified timestamp checks.
- Failure mode: NATS fallback causes message duplication.
  Prevention: message ids + de-dupe cache in both discovery handlers.
- Failure mode: manual add bypasses validation and injects invalid addresses.
  Prevention: strict parser with checksum/length validation and UI error states.
