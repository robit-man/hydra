# WO-12.1 NoClip Transport Marketplace Handler Activation

## Objective
Replace `MARKET_NOT_READY` transport stubs with live marketplace handlers so NoClip WebSocket transport can accept and emit marketplace catalog/status/quote/ticket events.

## Files To Touch
- `../noclip-unified/noclip-backend/packages/api/src/transport/plugin.ts`
  - `registerDefaultHandlers` stub block (`~71-160`), currently returns `MARKET_NOT_READY`.
- `../noclip-unified/noclip-backend/packages/api/src/transport/TransportServer.ts`
  - marketplace metric counters and recorder helpers (`~71-120`, `~435-470`).
- `../noclip-unified/noclip-backend/packages/api/src/services/marketplace.service.ts`
  - catalog ingest normalization + quote/settle flows (`~1174-1231`, `~2100+`).
- `../noclip-unified/noclip-backend/packages/api/src/transport/types.ts`
  - confirm canonical event constants for all marketplace events.

## Implementation Checklist
1. Remove stub response path in `plugin.ts` for marketplace events.
2. Route transport events to real marketplace service methods:
- `market.service.catalog` -> catalog ingest/merge.
- `market.service.status` -> status update path.
- `market.quote.request` / `market.quote.result` -> quote request/response flow.
- ticket/usage events -> verification + accounting hooks.
3. Enforce auth and payload validation parity with REST paths (same schema rules, same error codes).
4. Record transport marketplace metrics on success and failure.
5. Keep backward-compatible error envelope shape (interop contract fields + error code).

## Success Metrics
- Sending `market.service.catalog` over transport returns `status: ok` and persists catalog state.
- `MARKET_NOT_READY` is no longer returned for marketplace transport events.
- Quote/ticket events over transport produce equivalent outcomes to REST endpoints.
- Transport health payload reflects real marketplace activity timestamps/counters.

## Potential Failure Modes And Preventions
- Failure mode: transport event bypasses auth while REST is gated.
  Prevention: share auth/validation guard utilities with REST handlers.
- Failure mode: event handlers diverge from REST logic and produce inconsistent pricing/ledger states.
  Prevention: transport handlers call marketplace service methods directly (no duplicate business logic).
- Failure mode: malformed payloads crash websocket worker.
  Prevention: strict schema parse + structured `VALIDATION_ERROR` responses.
- Failure mode: replay or duplicate message processing.
  Prevention: require and enforce idempotency keys (`requestId`/`message_id`) where applicable.
