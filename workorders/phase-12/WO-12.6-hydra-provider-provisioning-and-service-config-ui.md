# WO-12.6 Hydra Provider Provisioning And Service Config UI

## Objective
Add explicit Hydra dashboard controls for provider identity and service publication configuration (pricing, visibility, transport candidates, health policy), replacing config-file-only workflows.

## Files To Touch
- `docs/index.html`
  - marketplace summary area (`~68-87`) expanded with provider/service config panel.
- `docs/style.css`
  - styling for provider editor, validation states, and save/apply controls.
- `docs/js/main.js`
  - bind new form controls to router control-plane APIs (`~600+`, `~1668+`).
- `service_router/router.py`
  - add/update control-plane endpoints for provider + `service_publication` mutation and persistence.
  - existing marketplace routes around `~10500-10613`.
- `service_router/router_config.json`
  - ensure editable fields map 1:1 with runtime config schema.

## Implementation Checklist
1. Add read/write APIs for marketplace provider + per-service publication configuration.
2. Add frontend form with:
- provider id/label/network/contact,
- per-service visibility/pricing/unit/capacity/tags,
- transport preference and include-unhealthy policy.
3. Add optimistic validation and explicit save/apply result feedback.
4. Ensure config persistence and runtime hot-apply are consistent.
5. Add import/export JSON controls for provider/service publication templates.

## Success Metrics
- Operator can fully configure provider metadata and service offerings in Hydra UI without manual file edits.
- Saving config updates router runtime and persists to `router_config.json`.
- Invalid values are blocked client-side and server-side with actionable errors.
- Catalog output reflects edited provider/service configuration immediately.

## Potential Failure Modes And Preventions
- Failure mode: UI saves invalid schema and breaks router startup.
  Prevention: shared schema validation on API path before persistence.
- Failure mode: runtime config differs from persisted config.
  Prevention: single source-of-truth apply function used by both save and reload.
- Failure mode: concurrent edits overwrite values.
  Prevention: config version/etag guard with conflict response.
- Failure mode: accidental publication of disabled services.
  Prevention: explicit enabled toggle and server-side default-off on invalid entry.
