# WO-12.4 Hydra Marketplace Discovery Event Pipeline

## Objective
Make Hydra discovery treat marketplace envelopes as first-class gossip events instead of only HTTP directory imports and target-bound bridge messages.

## Files To Touch
- `docs/js/peerDiscovery.js`
  - peer message type allowlist (`~1064-1073`).
  - shared inbound dispatcher (`~1075+`).
  - directory ingest helper (`~1555-1729`).
- `docs/js/nats.js`
  - discovery subject model (currently presence + DM only, `~173-176`).
- `docs/js/main.js`
  - directory polling client (`~606-758`) and ingestion wiring (`~769-820`).
- `docs/js/routerDiscovery.js`
  - ensure incoming catalog freshness and source annotations align with new gossip inputs.

## Implementation Checklist
1. Extend discovery message parsing to include:
- `market-service-catalog` / `market.service.catalog`,
- `market-service-status` / `market.service.status`.
2. Persist incoming market metadata directly into peer state (provider id/fingerprint, service health, selected transport).
3. Add optional NATS subject subscription path for market channels (not DM-only).
4. Keep HTTP directory import as fallback; annotate source priority (`nats-gossip`, `bridge-dm`, `http-directory`).
5. Add stale/replay guards using checksum + timestamp.

## Success Metrics
- Hydra peer list updates marketplace metadata when catalog/status gossip arrives over discovery.
- Market events no longer require pre-existing `NoClipBridge` watcher binding to be observed.
- HTTP directory polling can be disabled without losing discovery in NATS-enabled mode.
- Duplicate/old catalogs are dropped deterministically.

## Potential Failure Modes And Preventions
- Failure mode: market events collide with chat/ping peer protocol handlers.
  Prevention: explicit event namespace handling and non-overlapping type gates.
- Failure mode: stale event accepted due missing timestamp.
  Prevention: enforce timestamp fallback policy + checksum recency map.
- Failure mode: NATS subject flood overloads UI.
  Prevention: event rate limiter and coalesced UI updates.
- Failure mode: source precedence flaps between HTTP and gossip.
  Prevention: deterministic source ranking with freshness tie-breakers.

## Verification
1. `node --check docs/js/peerDiscovery.js docs/js/nats.js docs/js/main.js`
2. Inject synthetic market gossip envelopes and verify peer state updates in UI.
3. Disable HTTP directory polling and confirm providers remain discoverable via gossip.
4. Replay old checksum envelope and verify it is ignored.
