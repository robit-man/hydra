## Implementation Plan

1. Add a border symbol table inside `EnhancedUI` (thick corners/edges/intersections only).
2. Replace `_draw_box` with a new `_draw_panel_box(...)` using the symbol table for all panel renders.
3. Add `_halftone_char_for_x(x, width)`:
   - Normalize `ratio = x / (width - 1)`.
   - Use density ramp from heavy to light, e.g. `["Ō¢ł", "Ō¢ō", "Ō¢Æ", "Ō¢æ", "┬Ę"]`.
4. Apply halftone placement rules:
   - Top/bottom edges use per-cell halftone char by X ratio.
   - Vertical lines use the same per-column density.
   - Corners and intersections remain thick structural glyphs (not replaced by dots).
5. Replace old frame chrome (`_render_frame_chrome`) with the new primitive so every border is consistent.
6. Add a feature flag in `EnhancedUI` to disable halftone for fallback terminals.

## Success Metrics

- Every panel border in `main`, `config`, `stats`, `addressbook`, `ingress`, `egress`, `debug` uses the same renderer.
- Border density visibly transitions left-to-right at runtime.
- No mixed border styles (`ŌöÅ` in one panel, `ŌĢö` in another) unless explicitly configured.
- No curses write exceptions under terminal resize stress.

## Failure Modes + Preventive Controls

- Failure: Unicode width mismatch causes broken columns.
  - Prevention: restrict ramp to known single-cell glyphs and clamp all writes with `_safe_addstr`.
- Failure: Halftone chars overwrite title labels.
  - Prevention: draw border first, then titles/tabs as overlay.
- Failure: Terminal without extended glyph support looks corrupted.
  - Prevention: fallback mode using plain thick ASCII-compatible box set.
