# WO-06.2 Auth + Sensitive Field Handling

## Objective
Prevent sensitive values (seeds/passwords) from leaking via API responses, logs, and frontend exports.

## Files To Touch (Hydra)
- `service_router/router.py` control-plane JSON responses (`/health`, `/nkn/info`, `/services/snapshot`, `/api`)
- `service_router/router_config.json` secret fields (`seed_hex` values, lines 31-84)
- `docs/js/config.js` persisted frontend fields (lines 16-41)
- `docs/js/flows.js` export/import handling (line anchors to identify secret-bearing fields)

## Reference Anchors (Teleoperation)
- `/home/robit/Documents/repositories/Dropbear-Neck-Assembly/teleoperation/router/router.py` `nkn/info` seed exposure pattern (`seed_persisted` only, lines 3511-3519)
- `/home/robit/Documents/repositories/Dropbear-Neck-Assembly/docs/script.js` auth endpoint/password handling path (lines 2397-2432, 2468-2532)

## Implementation Checklist
1. Redact/omit secret fields in public API output.
2. Ensure frontend stored config excludes raw secrets by default.
3. Ensure flow export omits sensitive auth/seeds unless explicitly requested.
4. Add unit checks for redaction paths.

## Success Metrics
- No API route returns raw seed/password fields.
- Flow exports do not contain seed/password material.
- Logs avoid printing secret values.

## Potential Failure Modes And Preventions
- Failure mode: accidental secret serialization through debug dumps.
  Prevention: centralized redaction utility for all API payloads.
- Failure mode: frontend export regression reintroduces secrets.
  Prevention: schema-based export allowlist.
- Failure mode: partial redaction leaves identifiable substrings.
  Prevention: redact entire sensitive field, not partial masking.
